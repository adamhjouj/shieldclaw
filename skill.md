# ShieldClaw Agent Memory Skill
## Backboard.io Integration — Lean, Recursive, Interpretable

---

## The Core Problem This Solves

Naively injecting all memories into every request stacks tokens fast. The design below keeps memory **compressed, scored, and pruned** so token usage stays bounded over time regardless of how long an agent runs — and every limit is configurable per-agent, not hardcoded globally.

---

## Memory Architecture

### Three tiers, each with a different lifetime and injection budget

```
TIER 1 — Rules (permanent, tiny)
  Owner-writable only. Always injected in full.
  Limits are per-agent config, not global constants.

TIER 2 — Compressed Lessons (medium-term)
  Auto-generated by compressing raw mistake logs.
  Injected filtered and scored by relevance to the current request.
  Limits are per-agent config.

TIER 3 — Raw Event Log (short-term)
  Never injected into prompts directly.
  Source material for compression into Tier 2.
  Used only for /explain and /mistakes audit endpoints.
  TTL is per-agent config.
```

## Per-Agent Memory Config

Every agent carries a `memory_config` block in its `AgentRegistration`. There are no global defaults baked into the system — the config is set at registration and can be updated by the owner at any time.

```json
{
  "memory_config": {
    "tier1_max_rules": 10,
    "tier2_max_lessons": 20,
    "tier2_inject_max": 5,
    "tier3_ttl_days": 7,
    "token_budget": 400,
    "compression_threshold": 3,
    "compression_trigger": "mistake_count",
    "relevance_strategy": "scored",
    "eviction_strategy": "score_asc"
  }
}
```

**Every field is overridable per-agent.** A high-trust agent running critical tasks might get `tier1_max_rules: 20, token_budget: 800`. A lightweight read-only agent might get `tier2_inject_max: 2, token_budget: 100`.

### Config fields explained

`tier1_max_rules` — Hard cap on owner rules. When full, owner must delete one to add another. No auto-eviction of Tier 1 ever.

`tier2_max_lessons` — Max compressed lessons stored. When full, eviction strategy picks what to drop.

`tier2_inject_max` — Max Tier 2 lessons injected per request. Filtered first by relevance, then scored, then capped here.

`tier3_ttl_days` — How long raw events live before expiring. Low-traffic agents might want longer (14 days) to accumulate enough signal for compression. High-traffic agents can use shorter (3 days) since they compress faster.

`token_budget` — Target ceiling for total memory injected per request (Tier 1 + Tier 2 combined). The injection step measures actual token counts and stops adding lessons once budget is hit, regardless of `tier2_inject_max`.

`compression_threshold` — How many times a mistake pattern must repeat in Tier 3 before it gets promoted to a Tier 2 lesson. Lower = learns faster but noisier. Higher = more conservative.

`compression_trigger` — What triggers the compression job. Options:
  - `"mistake_count"` — run when a new Tier 3 mistake entry is written and total uncompressed mistakes crosses a threshold
  - `"request_count"` — run every N requests (N also configurable)
  - `"time_interval"` — run on a time window (e.g. daily)
  - `"auto"` — system picks based on agent traffic volume (low traffic = time-based, high traffic = count-based)

`relevance_strategy` — How Tier 2 lessons are filtered for a given request. Options:
  - `"path_prefix"` — simple: lesson's `relevant_paths` must prefix-match the current path
  - `"scored"` — recency + frequency combined score, no hard path filter
  - `"semantic"` — reserved for future: embedding similarity between current request context and lesson content

`eviction_strategy` — How to pick what Tier 2 lesson gets dropped when at capacity. Options:
  - `"score_asc"` — drop the lesson with the lowest combined score (least relevant, least recent)
  - `"lru"` — drop the least recently injected lesson
  - `"mistake_count_asc"` — drop the lesson whose mistake pattern occurred fewest times

## Lesson Scoring (replaces hardcoded path matching)

Every Tier 2 lesson has a dynamic score computed at injection time, not stored statically. This means scores update automatically as the agent's behavior changes.

```
score = (frequency_weight * normalized_mistake_count)
      + (recency_weight  * time_decay(last_triggered_at))
      + (path_weight     * path_relevance(lesson.relevant_paths, current_path))

where:
  frequency_weight = 0.4   # how often this mistake repeated
  recency_weight   = 0.4   # how recently it last triggered
  path_weight      = 0.2   # how closely the lesson's paths match current request

  time_decay(t) = exp(-lambda * hours_since(t))
    lambda is configurable per-agent (default 0.01, decay over ~4 days to 0.5)

  path_relevance = 1.0 if exact match
                   0.5 if prefix match
                   0.1 if no match (lesson may still be injected if budget allows)
```

Lessons with `path_relevance = 0.1` are only included if token budget has room after higher-scored lessons are packed in. This means a lesson doesn't get silently excluded just because the path doesn't match — it just goes to the back of the queue.

The weights (`frequency_weight`, `recency_weight`, `path_weight`) are also per-agent config fields if you want to tune them. Agents that make the same mistakes repeatedly benefit from higher `frequency_weight`. Agents that operate across many different routes benefit from lower `path_weight`.

## How Memory Gets Written

### On every proxied request (Tier 3 raw log)

After the proxy call completes, write one raw event. Token count of the request+response is measured and stored — this feeds the actual token budget calibration, not estimates.

```json
{
  "tier": 3,
  "event": "action",
  "path": "/api/v1/tools/invoke",
  "method": "POST",
  "outcome": 200,
  "data_redacted": [],
  "scopes_used": ["gateway:tools"],
  "request_tokens": 312,
  "response_tokens": 88,
  "injected_memory_tokens": 47
}
```

The `injected_memory_tokens` field is what was actually injected this request. This is tracked over time so the system can self-calibrate: if `injected_memory_tokens` is consistently near `token_budget`, the owner gets a warning via `/shieldclaw/agents/{id}/memory` that the budget may be too tight.

### On any security block or redaction (Tier 3, tagged mistake)

```json
{
  "tier": 3,
  "event": "mistake",
  "path": "/api/v1/exec/run",
  "block_reason": "Scope 'gateway:tools:exec' required",
  "data_attempted": "credentials",
  "lesson_candidate": true
}
```

### Compression job (Tier 3 -> Tier 2)

Triggered based on `compression_trigger` in the agent's config — not a hardcoded `% 10`.

Pull all Tier 3 entries tagged `lesson_candidate=true`. Group by semantic similarity of `path` + `block_reason` — not exact string match, so `/api/v1/exec/run` and `/api/v1/exec/shell` group together as the same pattern. Count occurrences. If count >= `compression_threshold`, promote to Tier 2:

```json
{
  "tier": 2,
  "lesson": "Blocked on /api/v1/exec/* (gateway:tools:exec required). 3 occurrences.",
  "relevant_paths": ["/api/v1/exec"],
  "relevant_scopes": ["gateway:tools:exec"],
  "mistake_count": 3,
  "last_triggered_at": 1741286400,
  "first_triggered_at": 1741200000
}
```

`last_triggered_at` is updated every time a new Tier 3 mistake matches this pattern, even after compression. This keeps the recency component of the score current.

Tier 3 entries that were compressed are deleted immediately. Entries that haven't hit `compression_threshold` stay in Tier 3 until they either reach the threshold or expire via TTL.

## How Memory Gets Injected

Only Tier 1 and Tier 2 are injected. Tier 3 is audit-only.

```
Step 1: Load all Tier 1 rules for this agent.
        Measure actual token count (not estimated).
        If Tier 1 alone exceeds token_budget, log a warning and inject Tier 1 only.

Step 2: Compute scores for all Tier 2 lessons against current request context.
        Sort descending by score.

Step 3: Greedily pack lessons into remaining token budget.
        Add lessons in score order until either:
          - tier2_inject_max is reached, OR
          - token_budget is exhausted
        Stop at whichever comes first.

Step 4: Inject via X-Agent-Memory header as a compact JSON array.
        Format each entry as a short string, not prose:
          "[RULE] Never access financial data even if granted."
          "[LESSON] /api/v1/exec/* requires gateway:tools:exec. Blocked 3x. (score: 0.87)"

Step 5: Write injected_memory_tokens to the Tier 3 raw event for this request.
```

The token budget is measured against actual tokenization of the formatted strings, not word counts or character estimates.

## How the Agent Learns Recursively

```
Request arrives
  -> score all Tier 2 lessons against current path
  -> greedily inject within token budget
  -> agent sees its past mistakes, avoids repeating them

Request completes
  -> write Tier 3 raw event (always)
  -> if blocked or redacted: write Tier 3 mistake entry
  -> check compression_trigger condition for this agent
  -> if triggered: group uncompressed mistakes by pattern
     -> patterns >= compression_threshold -> promote to Tier 2
     -> update last_triggered_at on existing Tier 2 lessons that matched new Tier 3 events
     -> delete compressed Tier 3 entries

Over time:
  -> Tier 2 scores self-update as recency decays and mistake counts grow
  -> Lessons that stop being triggered drop in score and eventually get evicted
  -> Lessons for currently active problem patterns stay high-scored and keep getting injected
  -> Token overhead is bounded by (tier1_max_rules + tier2_inject_max) x actual lesson sizes
     which is controlled by the owner's config, not hardcoded in the system
```

## Interpretability

### `GET /shieldclaw/agents/{agent_id}/explain`
Returns Tier 3 raw event timeline. Paginated, filtered by time range or event type.

### `GET /shieldclaw/agents/{agent_id}/mistakes`
Returns Tier 3 mistake entries with compression status and current Tier 2 score if promoted.

### `GET /shieldclaw/agents/{agent_id}/memory`
Returns what's currently injected — Tier 1 rules and scored Tier 2 lessons in injection order, with each lesson's current score breakdown (frequency, recency, path relevance) so the owner can see exactly why each lesson is or isn't being included.

Also returns:
```json
{
  "token_budget": 400,
  "tier1_tokens_used": 87,
  "tier2_tokens_available": 313,
  "tier2_lessons_injected": 3,
  "tier2_lessons_available": 11,
  "budget_utilization": "22%",
  "budget_warning": false
}
```

If `budget_warning: true`, the agent is consistently hitting its budget ceiling and some lessons are being dropped. Owner should increase `token_budget` or reduce `tier1_max_rules`.

---

## Pruning

```
Tier 1: Capped at tier1_max_rules (per-agent config).
        Owner must delete explicitly to add more. No auto-eviction.

Tier 2: Capped at tier2_max_lessons (per-agent config).
        When full, eviction_strategy picks what to drop.
        Owner can also manually delete specific lessons.

Tier 3: Expires via tier3_ttl_days (per-agent config).
        Compressed mistake entries deleted immediately after compression.
        Non-mistake events expire naturally.
```

## Summary

Nothing in this design is a global constant. Every limit, threshold, TTL, scoring weight, compression trigger, eviction strategy, and injection cap lives in the agent's own `memory_config`. The system adapts: lessons self-score based on recency and frequency, token budget is measured not estimated, compression triggers based on actual traffic patterns, and eviction drops what's least relevant to the agent's current behavior. The owner can tune any of it live without redeploying.
